# 行程规划方法论

## 输入收集

最少需要：出发/回程日期、到达/离开时间、机场/车站、酒店名和地址、一定要去/不要去的点。

最好补充：人数、年龄段、有无长辈小孩、餐饮预算、能否排队、能走多少路、偏好主题、行李和交通偏好、是否愿意早起。

## 判断顺序

1. 天数是否足够。
2. 酒店在哪，第一天和最后一天是否受机场/车站约束。
3. 用户明确讨厌什么，先排除。
4. 哪些点天气敏感。
5. 哪些点需要预约或票务确认。
6. 哪些点节假日高风险。
7. 最后再补餐厅、咖啡和备选。

## 选点原则

- 一天一个主区域。
- 一天只放一个重预约点。
- 第一天轻手，最后一天不跑远。
- 长距离点优先放在非节假日。
- 户外点标记天气敏感，并配附近室内备选。
- 用户明确讨厌的类型直接避开。

## 选餐原则

- 先保当天区域顺路，再给 2-3 个餐厅候选。
- 大众点评看口味、排队、踩雷、价格、区域和值不值得。
- 小红书看氛围、近期体验、拍照、软性提醒。
- 餐厅默认是补给点，不要为了高分店扭曲路线；只有预约餐、强目的餐、用户明确指定的店，才允许反过来影响路线。

## 执行步骤

1. 抽硬约束：到达时间、返程时间、航站楼/车站、酒店位置。
2. 把用户清单分成城内轻松组、预约组、远郊组、可路过组。
3. 主动删高风险点：天数不够的远郊、节假日爆点、短行程里放不进的天气敏感点。明说删了什么以及为什么删。
4. 按区域重组，减少换乘和折返。
5. 按当天主区域补餐厅、咖啡、酒吧，不让餐饮反向拖垮路线。
6. 查关键票务、营业时间、休馆日、机场/车站交通。
7. 输出结论、每日主区域、天气敏感点、餐饮候选、删点原因、提前订什么和返程安排。

## 信息源优先级

1. 官方景点官网、票务页、机场页、交通页。
2. 餐厅：大众点评 + 小红书。
3. 社媒种草和游记只补感觉，不能当营业时间或票务事实。

## 用户交互：四拍格式

需要用户决策时使用 Re-ground -> Simplify -> Recommend -> Options：

- Re-ground：告诉用户当前阶段，例如“行程规划，路线取舍”。
- Simplify：用大白话说清在问什么。
- Recommend：给一个明确推荐和理由。
- Options：给 2-4 个自解释选项。

关键原则：Smart skip 用户已给的信息；合并无依赖问题；非必要问题由 agent 代为默认；严格区分“计划会做”和“已经完成”。
