# Multi-User Travel Output Format

Keep outputs decision-first and transparent. Always separate recalled facts, user-provided facts, assumptions, and missing information.

## Multi-User Profile Summary

```md
参与人画像
- 用户 A（feishu_id: ...）：偏好：...｜粗粒度出发点：...｜硬约束：...
- 用户 B（feishu_id: ...）：偏好：...｜粗粒度出发点：...｜硬约束：...
- 用户 C（feishu_id: ...）：偏好：...｜粗粒度出发点：...｜硬约束：...

缺失信息
- 用户 X：缺少粗粒度出发点，当前不能准确计算通勤公平性。
- 用户 Y：没有召回到稳定偏好，按默认“适中人流、轻松节奏、广谱餐饮”处理。
```

## Conflict Explanation

```md
偏好冲突
- 冲突点：A 想人少，B 想热闹拍照。
- 处理：选择热闹商圈里的低排队/可预约店，晚上活动放在可进可退的街区。
- 备选：如果 A 更重视安静，切换到 ...；如果 B 更重视拍照，切换到 ...。
```

## Map Readiness Gate

Before offering or generating HTML, check whether the plan is stable enough:

```md
地图生成前确认
- 天气：已查/已按晴雨两套方案处理/待确认
- 住宿或集合点：已确认/按默认区域处理/待确认
- 出行方式：地铁/打车/自驾/步行强度已确认或有默认
- 食物：餐厅或餐饮区域已补齐，排队和预算风险已说明
- 美景/体验点：主体验点已确定，拍照/夜景/室内备选已处理
- 群成员偏好：关键偏好和硬禁忌已覆盖，冲突已有补救

结论：
A. 已稳定，可以生成地图
B. 还差 [1-2 个关键项]，建议先确认再生成
C. 用户明确要草稿，可生成“草稿地图”并标注待补齐项
```

如果选择 B，不要只说“请补充信息”。给用户一个最小决策问题，例如：

```md
这个版本的大方向已经顺了。为了避免地图刚生成又大改，我建议先确认一个点：住宿/集合点大概在哪个区域？
A. 用你们当前群里提到的位置
B. 我先按市中心/交通枢纽默认处理
C. 我们先不生成地图，只看文字方案
```

## Evidence Notes

Use short labels:

- `硬事实`：路线、地理位置、营业时间、天气、车次等已验证事实。
- `软信号`：点评评论标签、小红书体验、口碑描述、常识性拥挤判断。小红书链接和摘要只能说明近期氛围、拍照角度、排队体感、避坑提醒和适合人群，不能当作官方事实。
- `待验证`：日期未定、路线 API 未查、用户出发点缺失、实时状态不可得。

Never present soft signals or missing data as confirmed facts.
